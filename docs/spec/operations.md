# Operations

> How Vorpal is built, deployed, and run in production and development environments.

## CI/CD Pipeline

### Primary Workflow (`vorpal.yaml`)

Triggered on pull requests and pushes to `main` or tags. Uses concurrency groups to cancel
superseded runs for the same PR or ref.

**Pipeline stages (sequential, with matrix expansion):**

1. **vendor** — Runs on a 4-runner matrix (macOS x86_64, macOS aarch64, Ubuntu x86_64, Ubuntu
   aarch64). Checks out code, restores cached `target/` and `vendor/` directories keyed by
   `{arch}-{os}-{Cargo.lock hash}`, runs `./script/dev.sh` for environment bootstrap, vendors
   Cargo dependencies, runs `cargo check --offline --release`, then saves caches.

2. **code-quality** — Runs on `macos-latest` only (after vendor). Restores caches, runs
   `cargo fmt --all --check` and `cargo clippy --offline --release -- --deny warnings`.

3. **build** — Runs on the same 4-runner matrix (after code-quality). Executes
   `cargo build --offline --release`, `cargo test --offline --release`, and packages a
   distribution tarball (`dist/vorpal-{arch}-{os}.tar.gz`). Uploads the tarball as a GitHub
   Actions artifact.

4. **test** — Runs on the same 4-runner matrix (after build). Downloads the built distribution,
   extracts it, configures Vorpal via `ALT-F4-LLC/setup-vorpal-action@main` with an S3 registry
   backend (`altf4llc-vorpal-registry` bucket), then runs end-to-end parity tests. These tests
   build artifacts using the Rust SDK (`Vorpal.toml`), Go SDK (`Vorpal.go.toml`), and TypeScript
   SDK (`Vorpal.ts.toml`), verifying that all three produce identical artifact digests for each
   artifact name. Artifacts tested: `vorpal`, `vorpal-container-image` (Linux only), `vorpal-job`,
   `vorpal-process`, `vorpal-shell`, `vorpal-user`. Uploads `Vorpal.lock` as an artifact.

5. **container-image** — Tag pushes only, Linux runners only (after test). Builds a container
   image via `vorpal build --path "vorpal-container-image"`, loads it into Docker, tags it as
   `docker.io/altf4llc/vorpal:{tag}-{amd64|arm64}`, and pushes to Docker Hub.

6. **container-image-manifest** — Tag pushes only (after container-image). Creates and pushes a
   multi-arch Docker manifest combining `amd64` and `arm64` images under
   `docker.io/altf4llc/vorpal:{tag}`.

7. **release** — Tag pushes only (after test). Downloads all 4 platform distribution tarballs,
   creates a GitHub Release (prerelease) with the tag name, attaches all tarballs, and generates
   build provenance attestations via `actions/attest-build-provenance@v3`.

### Nightly Workflow (`vorpal-nightly.yaml`)

Runs on a daily cron schedule (`0 8 * * *`) and supports manual dispatch. Uses a GitHub App token
to delete any existing `nightly` release and tag, then recreates the `nightly` tag pointing at the
current `main` branch HEAD SHA. This triggers the primary workflow to produce a fresh nightly
release.

### Setup Action

The `ALT-F4-LLC/setup-vorpal-action@main` GitHub Action is used in CI to configure the Vorpal
runtime. It accepts parameters for registry backend type (`s3`), S3 bucket name, and whether to
use a local build. It injects AWS credentials via environment variables.

### Caching Strategy

- **Cargo target directory**: Keyed by `{arch}-{os}-{Cargo.lock hash}-target`. Saves/restores the
  `target/` directory to avoid redundant compilation across jobs.
- **Vendor directory**: Keyed by `{arch}-{os}-{Cargo.lock hash}-vendor`. Caches vendored crate
  sources for offline builds.
- Cache is saved after the vendor stage and restored in code-quality, build, and test stages.

## Dependency Management

### Renovate

Configured via `.github/renovate.json`. Extends `config:base` with semantic commit type `chore`.
Lock file maintenance runs weekly. Ignores paths under `cli/src/command/template/**` to avoid
modifying project templates.

### Cargo Vendoring

Dependencies are vendored into the `vendor/` directory via `cargo vendor --versioned-dirs`. A
local `.cargo/config.toml` is generated by the `make .cargo` target to redirect crate resolution
to the vendored sources. This enables fully offline builds in CI.

### Vorpal Lock File

`Vorpal.lock` tracks source artifact URLs, digests (SHA-256), platform, and
include/exclude filters for all SDK toolchain dependencies (Rust toolchain, Go, Node.js, Bun,
protoc, etc.). This file is generated during the test phase and uploaded as a CI artifact.

## Development Environment

### Bootstrap Script (`script/dev.sh`)

Entry point for development environment setup. Detects the host OS and installs platform-specific
dependencies:

- **Debian/Ubuntu**: `bubblewrap build-essential ca-certificates curl jq rsync unzip docker`
  (via `script/dev/debian.sh`)
- **Arch Linux**: `docker bubblewrap ca-certificates curl unzip` (via `script/dev/arch.sh`)

Then installs the following tools into a local `.env/bin` directory:

| Tool | Script | Version |
|---|---|---|
| xz | `script/dev/xz.sh` | 5.8.2 (built from source) |
| Amber | `script/dev/amber.sh` | 0.5.1 |
| Rustup/Cargo | `script/dev/rustup.sh` | Via rustup (toolchain from `rust-toolchain.toml`: 1.93.1) |
| Protoc | `script/dev/protoc.sh` | 34.0 |
| Terraform | `script/dev/terraform.sh` | 1.14.6 |

All tool scripts are idempotent — they skip installation if the binary already exists.

### Makefile

The `makefile` provides the primary development interface:

| Target | Description |
|---|---|
| `build` (default) | `cargo build` |
| `check` | `cargo check` |
| `format` | `cargo fmt --all --check` |
| `lint` | `cargo clippy -- --deny warnings` |
| `test` | `cargo test` |
| `dist` | Package binary into `dist/vorpal-{arch}-{os}.tar.gz` |
| `vendor` | `cargo vendor --versioned-dirs` |
| `clean` | Remove `target/`, `.cargo/`, `dist/`, `vendor/` |
| `generate` | Regenerate Go and TypeScript gRPC stubs from proto files |
| `vorpal` | Build artifact via `cargo run -- build` |
| `vorpal-start` | Start services via `cargo run -- system services start` |

The `TARGET` variable controls build profile (`debug` by default, `release` for CI).

### Lima Virtual Machine

The project includes a Lima VM configuration (`lima.yaml`) for running Linux builds on macOS. Uses
Debian 12 (Bookworm) cloud images with pinned digests for both x86_64 and aarch64. Makefile
targets:

- `lima` — Create and provision a Debian VM with the full dev toolchain
- `lima-sync` — Sync project files into the VM and rebuild
- `lima-vorpal` — Run `vorpal build` inside the VM
- `lima-vorpal-start` — Start Vorpal services inside the VM

Defaults: 8 CPUs, 8 GB RAM, 100 GB disk.

### Docker Compose

`docker-compose.yaml` runs a local Keycloak instance for auth development:

```yaml
services:
  keycloak:
    image: quay.io/keycloak/keycloak:26.5.4
    command: start-dev
    ports: 127.0.0.1:8080:8080
```

Bootstrap credentials: `admin`/`password`.

## Installation & Service Management

### Install Script (`script/install.sh`)

End-user installation script. Downloads the `nightly` release tarball from GitHub to
`~/.vorpal/bin/`, creates the Vorpal data directory structure, generates TLS keypairs, and
configures system service management.

**Data directory layout** (`/var/lib/vorpal/`):

```
/var/lib/vorpal/
  key/           # TLS keys, credentials
  log/           # Service logs (macOS only)
  sandbox/       # Build sandboxes (UUID-named)
  store/
    artifact/
      alias/     # Named artifact references
      archive/   # Compressed artifact archives (.tar.zst)
      config/    # Artifact configuration (.json)
      output/    # Built artifact outputs
```

### macOS: LaunchAgent

Creates `~/Library/LaunchAgents/com.altf4llc.vorpal.plist`:
- Runs `vorpal system services start`
- Starts at login (`RunAtLoad: true`)
- Auto-restarts on crash (`KeepAlive: true`)
- Logs to `/var/lib/vorpal/log/services.log`

### Linux: systemd

Creates `/etc/systemd/system/vorpal.service`:
- Type: `simple`
- Runs `vorpal system services start`
- Restart: `always` (5-second delay)
- Enabled and started via `systemctl`

## Service Runtime

### Service Architecture

The Vorpal server is a single binary running multiple gRPC services. Default service set:
`agent,registry,worker` (configurable via `--services`).

**Transport modes:**
- Unix domain socket (default): `/var/lib/vorpal/vorpal.sock` (overridable via
  `VORPAL_SOCKET_PATH`)
- TCP with optional TLS: `--port <port>` (default TLS port: 23151)

**Services:**

| Service | Proto | Purpose |
|---|---|---|
| Agent | `agent.proto` | Coordinates artifact builds |
| Archive | `archive.proto` | Stores/retrieves compressed artifact archives |
| Artifact | `artifact.proto` | Manages artifact metadata and configs |
| Worker | `worker.proto` | Executes build steps |

### Health Checks

gRPC health checking is available via `--health-check` flag. Uses the `tonic-health` crate
implementing the standard gRPC health protocol. Runs on a separate plaintext TCP listener
(default port 23152) even when the main listener uses TLS or UDS. Each service registers its
health status individually.

### Graceful Shutdown

The server handles both `SIGINT` (Ctrl+C) and `SIGTERM` signals. On shutdown:
- Stops accepting new connections
- Cleans up the Unix socket file (if UDS mode)
- Advisory lock file is released automatically on process exit

### Process Locking

Uses `fs4` advisory file locking (`vorpal.lock` in the Vorpal root) to prevent multiple instances.
Before binding the UDS, checks if an existing socket is alive by attempting a connection — stale
sockets are cleaned up, live sockets cause a startup error.

### Registry Backend

Two backends for artifact storage:
- **local** (default): Stores artifacts on the local filesystem under `/var/lib/vorpal/store/`
- **s3**: Stores artifacts in an AWS S3 bucket (requires `--registry-backend-s3-bucket`)

### Authentication (Optional)

When `--issuer` is provided, the registry (Archive + Artifact) and Worker services are protected
by OIDC token validation. Uses JWKS discovery from the issuer URL to validate access tokens.
Worker service supports token exchange for accessing Archive and Artifact services on behalf of
users.

## Infrastructure (Terraform)

The `terraform/` directory defines AWS infrastructure for a development/CI worker fleet.

### Workers Module (`terraform/module/workers/`)

Provisions EC2 instances across four platform targets:

| Instance | Type | AMI | Notes |
|---|---|---|---|
| Registry (aarch64-linux) | `t4g.large` | Ubuntu 24.04 arm64 | Always created |
| Worker aarch64-linux | `t4g.large` | Ubuntu 24.04 arm64 | Always created |
| Worker x86_64-linux | `t3a.large` | Ubuntu 24.04 amd64 | Always created |
| Worker aarch64-darwin | `mac2.metal` | macOS Sequoia arm64 | Conditional (`create_mac_instances`) |
| Worker x86_64-darwin | `mac1.metal` | macOS Sequoia x86_64 | Conditional (`create_mac_instances`) |

All instances:
- 100 GB root volume
- Public IP with SSH access (configurable ingress CIDR via `ssh_ingress_cidr`)
- Shared VPC (`10.42.0.0/16`) in a single AZ
- SSH key pair managed via Terraform, private key stored in AWS SSM Parameter Store

### Keycloak Module (`terraform/module/keycloak/`)

Configures a Keycloak realm (`vorpal`) with:
- **Clients**: `cli` (public, device auth), `archive` (confidential), `artifact` (confidential),
  `worker` (confidential, service account)
- **Roles**: Per-client RBAC (`archive:check/push/pull`, `artifact:get/get-alias/store`,
  `worker:build-artifact`)
- **Scopes**: Audience-based scopes with protocol mappers for token exchange
- **Users**: Default `admin` user for development

### Providers

- AWS (`hashicorp/aws` v6.34.0)
- Keycloak (`keycloak/keycloak` v5.6.0)

## Logging & Observability

### Current State

- **Structured logging**: Uses the `tracing` crate with `tracing-subscriber`. Logs to stderr.
  Log level is configurable via `--level` (default: `INFO`). Debug/trace levels add file and
  line number information.
- **Service log output**: On macOS, the LaunchAgent redirects stdout/stderr to
  `/var/lib/vorpal/log/services.log`. On Linux, systemd captures logs via journald.
- **No centralized logging, metrics, or distributed tracing** is configured. There are no
  Prometheus endpoints, OpenTelemetry collectors, or APM integrations.

### Gaps

- No application-level metrics (request counts, latencies, error rates)
- No distributed tracing (spans, trace IDs across services)
- No alerting or monitoring dashboards
- No log rotation configured for the macOS LaunchAgent log file
- No structured error reporting or crash analytics

## Release Process

### Tagged Releases

1. Create and push a Git tag (e.g., `v0.1.0`)
2. CI pipeline runs the full vendor → code-quality → build → test → release flow
3. Release job creates a GitHub Release (marked as prerelease) with:
   - Distribution tarballs for all 4 platforms
   - Build provenance attestations (SLSA)
4. Container image job builds and pushes multi-arch Docker images to Docker Hub

### Nightly Releases

- Automated daily at 08:00 UTC via cron
- Deletes the previous `nightly` tag/release, recreates it at `main` HEAD
- Triggers the same full CI pipeline as tagged releases
- The install script defaults to downloading the `nightly` release

### Rollback

There is no formal rollback procedure documented. Options available:
- **Binary rollback**: Download a previous release tarball from GitHub Releases
- **Container rollback**: Pull a previous tagged image from Docker Hub
- **Service restart**: `launchctl`/`systemctl` commands depending on platform
- **Terraform**: Standard `terraform plan`/`apply` with previous state
